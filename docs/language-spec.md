# Kotoba Quest 言語仕様書

最終更新: 2026-05-09

---

## 1. 目的

- 本書は、Kotoba Quest で使用する**独自プログラミング言語の構文仕様**を整理するためのドキュメントである
- 要件定義書ではゲーム全体の目的や難易度を扱い、本書では**コードとして何をどう書けるか**を扱う
- パーサー実装、エディタ支援、実行基盤、ローカルヒント、AI ナビゲーターの前提となる仕様をまとめる

---

## 2. スコープ

- 本書では以下を扱う
- キーワードや文の形
- ブロック構造
- 変数、繰り返し、条件分岐などの基本構文
- 魔法、属性、合体魔法の記法
- プレイヤーコードと NPC コードの関係
- パーサーとエディタ実装の前提

- 本書では以下は詳細に扱わない
- ダメージ計算式
- 属性相性の数値バランス
- 敵 AI の挙動
- ステージごとの難易度調整

---

## 3. 言語コンセプト

- **日本語で Python のように書ける独自言語**とする
- 漢字・ひらがなをキーワードとして使用し、英語が苦手な小学生でも直感的に書けることを重視する
- プログラムは**上から下へ順に実行される**ことを基本とする
- 文法は、学習順である `順次実行 -> 繰り返し -> 変数 -> 条件分岐` に沿って段階的に理解できることを重視する
- 条件分岐は、常時自由に使う一般機能というより、**特定の敵ギミックに対応するための構文**として扱う
- プレイヤー用コードと NPC 用コードは別領域で扱うが、言語仕様の基本は共通とする

---

## 4. 現時点で確定している前提

### 4-1. 実行モデル

- バトルは**事前プログラミング型（ループ駆動）**であり、プレイヤーはバトル開始前にコードを記述する
- コードを実行すると、その内容に基づいて戦闘が再生される
- バトル中にリアルタイムでコードを書き換えることはしない
- コードは**上から順に評価・実行**される
- プレイヤーはメインループを自分で書き、そのループの**各繰り返し（ラウンド）**がバトルの1コマとなる
- ループ内のプレイヤー行動が終わると、敵が1回行動する
- 各ラウンド開始時に、変数のリセットと MP 回復が行われる
- 単属性魔法は `魔法(魔法名)` 命令で実行する
- 同一ラウンド内で、必要な種類の `魔法()` 命令が3種類以上実行され、かつ実行時点で必要 MP が足りている場合に**合体魔法**が発動する
- 必要な属性が揃っていても、**実行時点で MP が不足している場合はその時点で失敗**する

### 4-2. 学習に必要な構文カテゴリ

| 学習テーマ | 言語として必要になる要素 | 主に扱うステージ |
|------------|--------------------------|------------------|
| 順次実行・ループ駆動 | メインループ（`繰り返す(〜 あいだ):`）と基本命令 | ステージ1 |
| 繰り返し | ラウンド内で同じ行動をまとめる回数指定ループ（`繰り返す(N):`） | ステージ2 |
| 変数 | 値に名前を付けて扱う構文 | ステージ3 |
| 条件分岐 | 特定の敵ギミックに対して行動を切り替える構文 | ステージ4 |
| 合体魔法 | 複数属性を同一ラウンド内で組み合わせる記法 | ステージ4以降 |
| 相互参照 | プレイヤーと NPC が互いの値を参照する記法 | ステージ3以降 |

### 4-3. 魔法名と属性の前提

- 単属性魔法は、属性名をそのまま書く方式ではなく、**個別魔法名**を持つ方式を採用する
- 現時点の基本単属性魔法名は以下とする

| 属性 | 単属性魔法名 |
|------|--------------|
| 火 | フレイム |
| 水 | アクア |
| 雷 | スパーク |
| 氷 | フロスト |
| 風 | ゲイル |

- 属性の複合は **3種類以上のみ** を扱う
- **2属性のみの複合魔法は初期仕様では採用しない**
- 3属性以上を組み合わせた魔法は、合体魔法として扱う
- 単属性魔法の基本実行形は、`使用(フレイム)` のような記法を前提とする
- 変数で合体魔法の構成要素をまとめた簡略表現を用いる場合は、**消費MPを少し軽くする代わりに威力も少し下げる**前提とする

### 4-4. プレイヤーコードと NPC コード

- プレイヤーコードと NPC コードは**別々のコード領域**として扱う
- 両者は互いの値を**参照可能**とする
- 相互参照は、実装の複雑化を避けるため、**読み取り専用**を基本とする
- 相手の値を参照する時は、**ターン開始時点の値**を読む前提とする
- ターン中に相互参照が循環して実行順依存にならないようにする

### 4-5. 変数の寿命

- 変数は**ループの各繰り返し（ラウンド）開始時にリセット**する
- 変数の値は次のラウンドへ持ち越さず、毎ラウンド新しく評価する
- 変数は長期保存ではなく、**そのラウンド内で処理を整理したり簡略化したりするための仕組み**として扱う

### 4-6. 実装前提

- パーサーの正本は **Ohm.js** とする
- エディタは **CodeMirror 6** を使用するが、正誤判定は CodeMirror 独自実装ではなく Ohm.js の仕様を正とする
- パーサーは、将来的に**構文エラー**と**実行時エラー**を区別して扱えることが望ましい

## 5. 確定した構文仕様

本章は確定済みの構文仕様を記述する。

### 5-1. プログラム全体

- **文の区切り**: 改行ベース（1行1命令）。終端記号は不要
- **ブロック構造**: インデント型（Python 式）。スペース 2 個を基本とする
- **ブロック開始マーカー**: `:` をブロック先頭行の末尾に付ける
- **コメント**: 初期実装では扱わない

```
繰り返す(3):
  攻撃()
  魔法(フレイム)
```

### 5-2. 基本命令

すべての行動を**関数呼び出し形式**で統一する。プレイヤーは魔法のみで戦うため、物理攻撃コマンドは持たない。

| 命令 | 記法 | 説明 |
|------|------|------|
| 防御 | `防御()` | そのラウンドの被ダメージを -10 軽減する |
| 回復 | `回復()` | HP を最大HPの30%（30）回復する |
| 待機 | `待機()` | 行動をスキップして MP を温存する |
| 魔法 | `魔法(魔法名)` | 指定した単属性魔法を使用する |

```
防御()
回復()
魔法(フレイム)
魔法(アクア)
```

### 5-3. 値と変数

- **変数宣言**: 宣言不要。代入した時点で変数として扱われる
- **代入記法**: `変数名 = 値`（イコール記号）
- **変数名に使える文字**: ひらがな・カタカナ・漢字・英数字
- **寿命**: ターンごとにリセット。次のターンへ引き継がない

```
こうげきかいすう = 3
まほう = フレイム
魔法(まほう)
```

### 5-4. 繰り返し

繰り返し構文は2種類ある。

#### 条件付きループ（メインバトルループに使用）

- **記法**: `繰り返す(条件 あいだ):`
- 条件が真である間、ブロックを繰り返す
- バトルの外側ループとして使う

```
繰り返す(敵が生きている あいだ):
  もし 自分のMP が 80 以上 ならば:
    魔法(フレイム)
    魔法(アクア)
    魔法(スパーク)
  そうでなければ:
    攻撃()
```

条件には `ゲーム状態参照`（`敵のHP`, `自分のMP` 等）と比較演算子が使える。

#### 回数指定ループ（ラウンド内の繰り返し行動に使用）

- **記法**: `繰り返す(N):`（N は繰り返し回数の整数）
- ラウンド内で同じ命令を複数回まとめるために使う（ステージ2の学習テーマ）

```
繰り返す(敵が生きている あいだ):
  繰り返す(3):
    魔法(アクア)
```

- **ネスト**: メインループの内側に回数指定ループをネストする使い方を基本とする

### 5-5. 条件分岐

- **if に相当する記法**: `もし 条件 ならば:`
- **elif に相当する記法**: `そうでなければ もし 条件 ならば:`
- **else に相当する記法**: `そうでなければ:`
- **ブロック**: インデントで囲んだ行が対象

```
もし 敵が火状態 ならば:
  魔法(フレイム)
そうでなければ もし 敵が水状態 ならば:
  魔法(アクア)
そうでなければ もし 敵が雷状態 ならば:
  魔法(スパーク)
そうでなければ もし 敵が氷状態 ならば:
  魔法(フロスト)
そうでなければ:
  魔法(ゲイル)
```

#### 比較演算子（全て日本語）

| 記法 | 意味 |
|------|------|
| `より小さい` | `<` |
| `より大きい` | `>` |
| `以上` | `>=` |
| `以下` | `<=` |
| `と等しい` | `==` |
| `と等しくない` | `!=` |

#### 論理演算子（全て日本語）

| 記法 | 意味 |
|------|------|
| `かつ` | `and` |
| `または` | `or` |
| `ではない` | `not` |

#### 条件に使用できるゲーム状態参照

| 参照 | 意味 |
|------|------|
| `敵のHP` | 対象となる敵の現在 HP |
| `自分のHP` | プレイヤーの現在 HP |
| `自分のMP` | プレイヤーの現在 MP |
| `敵が生きている` | 敵の HP が 0 より大きい（メインループ条件に使用） |
| `敵が火状態` | 敵の現在状態が火（ステージ4以降） |
| `敵が水状態` | 敵の現在状態が水（ステージ4以降） |
| `敵が雷状態` | 敵の現在状態が雷（ステージ4以降） |
| `敵が氷状態` | 敵の現在状態が氷（ステージ4以降） |
| `敵が風状態` | 敵の現在状態が風（ステージ4以降） |

```
もし 自分のMP が 80 以上 かつ 敵のHP が 50 より小さい ならば:
  魔法(フレイム)
  魔法(アクア)
  魔法(スパーク)
そうでなければ:
  攻撃()
```

### 5-6. 魔法と属性

#### 単属性魔法

- **実行記法**: `魔法(魔法名)`
- **魔法名の一覧**:

| 属性 | 魔法名 |
|------|--------|
| 火 | フレイム |
| 水 | アクア |
| 雷 | スパーク |
| 氷 | フロスト |
| 風 | ゲイル |

#### 合体魔法

- **発動判定タイミング**: **ラウンド終了時にまとめて判定**する
- **発動条件**: 同一ラウンド内で **3種類以上の異なる属性** の `魔法()` 命令が実行され、かつ必要 MP が足りている場合に自動発動する
- **属性の重複**: 同一属性を何回使っても **1種類としてカウント**する
- **2属性合体は採用しない**
- **命名**: 組み合わせごとに固有名を持つ（合計16種）。名前はコンテンツ設計フェーズで確定する
- **威力**: 組み合わせによって異なる（数値はバランス調整フェーズで確定する）

| 属性数 | 種類数 | 消費 MP |
|--------|--------|---------|
| 3属性 | 10種 | 80 |
| 4属性 | 5種 | 100 |
| 5属性 | 1種 | 120 |

#### 通常版と省エネ版

`魔法()` に**直接魔法名を書く**のが通常版、**変数名を渡す**のが省エネ版。

```
# 通常版（フル威力・フルコスト）
魔法(フレイム)
魔法(アクア)
魔法(スパーク)

# 省エネ版（MP 安め・威力低め）
わざ1 = フレイム
わざ2 = アクア
わざ3 = スパーク
魔法(わざ1)
魔法(わざ2)
魔法(わざ3)
```

- 省エネ版の具体的な MP 軽減量・威力低下量は未確定（バランス調整フェーズで確定する）
- パーサーは引数が識別子（変数）かリテラル（魔法名）かで通常版 / 省エネ版を区別する

### 5-7. ターゲット指定

- **記法**: `敵[番号]へ 命令()` の形式（対象を先に書く）
- **指定なし**: 先頭（最前列）の敵を自動対象とする
- **全体指定**: 初期実装では扱わない

```
敵[1番目]へ 魔法(フレイム)
敵[2番目]へ 魔法(アクア)
魔法(スパーク)
```

対象を指定しない場合は先頭（最前列）の敵を自動対象とする。

### 5-8. プレイヤー / NPC 相互参照

- **参照記法**: `プレイヤー.変数名` / `なかま.変数名`
- **参照可能対象**: 変数のみ（HP / MP / 状態の相互参照は未確定）
- **参照タイミング**: ターン開始時点の値を読む（循環依存を防ぐ）
- **アクセス制限**: 読み取り専用（代入不可）

NPC コード:

```
まほう = フレイム
```

プレイヤーコード:

```
もし なかま.まほう と等しい フレイム ならば:
  魔法(アクア)
```

## 6. 仕様決定の方針

- ステージ1〜4を成立させる**最小構文**を優先する
- 関数定義や高度な抽象化は初期実装では入れない
- パーサー実装しやすさより、**小学生にとって読みやすく直感的であること**を優先する
- ローカルヒントや AI ナビゲーターが説明しやすいよう、曖昧すぎる表現は避ける
