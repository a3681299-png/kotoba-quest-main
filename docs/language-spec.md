# Kotoba Quest 言語仕様書

最終更新: 2026-05-05

---

## 1. 目的

- 本書は、Kotoba Quest で使用する**独自プログラミング言語の構文仕様**を整理するためのドキュメントである
- 要件定義書ではゲーム全体の目的や難易度を扱い、本書では**コードとして何をどう書けるか**を扱う
- パーサー実装、エディタ支援、実行基盤、ローカルヒント、AI ナビゲーターの前提となる仕様をまとめる

---

## 2. スコープ

- 本書では以下を扱う
- キーワードや文の形
- ブロック構造
- 変数、繰り返し、条件分岐などの基本構文
- 魔法、属性、合体魔法の記法
- プレイヤーコードと NPC コードの関係
- パーサーとエディタ実装の前提

- 本書では以下は詳細に扱わない
- ダメージ計算式
- 属性相性の数値バランス
- 敵 AI の挙動
- ステージごとの難易度調整

---

## 3. 言語コンセプト

- **日本語で Python のように書ける独自言語**とする
- 漢字・ひらがなをキーワードとして使用し、英語が苦手な小学生でも直感的に書けることを重視する
- プログラムは**上から下へ順に実行される**ことを基本とする
- 文法は、学習順である `順次実行 -> 繰り返し -> 変数 -> 条件分岐` に沿って段階的に理解できることを重視する
- 条件分岐は、常時自由に使う一般機能というより、**特定の敵ギミックに対応するための構文**として扱う
- プレイヤー用コードと NPC 用コードは別領域で扱うが、言語仕様の基本は共通とする

---

## 4. 現時点で確定している前提

### 4-1. 実行モデル

- バトルは**事前プログラミング型**であり、プレイヤーはバトル開始前にコードを記述する
- コードを実行すると、その内容に基づいて戦闘が再生される
- バトル中にリアルタイムでコードを書き換えることはしない
- コードは、原則として**上から順に評価・実行**される
- 単属性魔法は `使用(...)` のような命令で実行する前提とする
- 同一ターン内で、必要な種類の `使用(...)` 命令が3種類以上実行され、かつ実行時点で必要MPが足りている場合に**合体魔法**が発動する
- 必要な属性が揃っていても、**実行時点でMPが不足している場合はその時点で失敗**する

### 4-2. 学習に必要な構文カテゴリ

| 学習テーマ | 言語として必要になる要素 | 主に扱うステージ |
|------------|--------------------------|------------------|
| 順次実行 | 行動を順番に書く基本命令 | ステージ1 |
| 繰り返し | 同じ行動をまとめる構文 | ステージ2 |
| 変数 | 値に名前を付けて扱う構文 | ステージ3 |
| 条件分岐 | 特定の敵ギミックに対して行動を切り替える構文 | ステージ4 |
| 合体魔法 | 複数属性を組み合わせる記法 | ステージ4以降 |
| 相互参照 | プレイヤーと NPC が互いの値を参照する記法 | ステージ3以降 |

### 4-3. 魔法名と属性の前提

- 単属性魔法は、属性名をそのまま書く方式ではなく、**個別魔法名**を持つ方式を採用する
- 現時点の基本単属性魔法名は以下とする

| 属性 | 単属性魔法名 |
|------|--------------|
| 火 | フレイム |
| 水 | アクア |
| 雷 | スパーク |
| 氷 | フロスト |
| 風 | ゲイル |

- 属性の複合は **3種類以上のみ** を扱う
- **2属性のみの複合魔法は初期仕様では採用しない**
- 3属性以上を組み合わせた魔法は、合体魔法として扱う
- 単属性魔法の基本実行形は、`使用(フレイム)` のような記法を前提とする
- 変数で合体魔法の構成要素をまとめた簡略表現を用いる場合は、**消費MPを少し軽くする代わりに威力も少し下げる**前提とする

### 4-4. プレイヤーコードと NPC コード

- プレイヤーコードと NPC コードは**別々のコード領域**として扱う
- 両者は互いの値を**参照可能**とする
- 相互参照は、実装の複雑化を避けるため、**読み取り専用**を基本とする
- 相手の値を参照する時は、**ターン開始時点の値**を読む前提とする
- ターン中に相互参照が循環して実行順依存にならないようにする

### 4-5. 変数の寿命

- 変数は**ターンごとにリセット**する
- 変数の値は次のターンへ持ち越さず、毎ターン新しく評価する
- 変数は長期保存ではなく、**そのターン内で処理を整理したり簡略化したりするための仕組み**として扱う

### 4-6. 実装前提

- パーサーの正本は **Ohm.js** とする
- エディタは **CodeMirror 6** を使用するが、正誤判定は CodeMirror 独自実装ではなく Ohm.js の仕様を正とする
- パーサーは、将来的に**構文エラー**と**実行時エラー**を区別して扱えることが望ましい

## 5. 構文仕様の整理対象

本章は、今後具体的な記法を決めていく対象を整理したものである。

### 5-1. プログラム全体

- 1つのコード領域を何と呼ぶか
- 1行ごとの文の区切り方
- 改行の扱い
- 空白の扱い
- コメント記法の有無

### 5-2. 基本命令

- 攻撃
- 防御
- 回復
- 単属性魔法の使用
- 待機や MP を溜める行動の要否

### 5-3. 値と変数

- 数値リテラルの表記
- 変数宣言の有無
- 代入の書き方
- 変数名に使える文字
- 予約語の扱い
- ターンリセットを記法上どう意識させるか

### 5-4. 繰り返し

- 繰り返し回数の指定方法
- ブロックの開始と終了をどう表現するか
- ネストを許可するか

### 5-5. 条件分岐

- `もし`
- `そうでなければ`
- 比較演算子
- 論理演算子
- 敵状態、HP、MP、属性状態など何を条件として書けるか
- 条件分岐を必要とする敵ギミックの対象範囲

### 5-6. 魔法と属性

- 単属性魔法名をどう書くか
- `使用(...)` の最終記法
- 3属性以上の複数属性をどう組み合わせるか
- 3属性以上を合体魔法としてどう表現するか
- 属性名の正式表記
- 個別魔法名と属性の対応をどう扱うか
- 変数でまとめた場合の省エネ版合体魔法をどう表現するか
- MP 消費と記法の関係

### 5-7. ターゲット指定

- 単体対象の指定方法
- 複数対象や全体対象の扱い
- 敵 / 自分 / 仲間の区別
- 優先順位指定や条件付き対象指定の要否

### 5-8. プレイヤー / NPC 相互参照

- `プレイヤー.変数名` / `なかま.変数名` のような名前空間形式にするか
- 参照可能な対象を変数だけにするか、HP / MP / 状態も含めるか
- 読み取り専用であることを文法上どう表現するか

---

## 6. 仕様決定の進め方

構文仕様は、以下の順で固めることを推奨する。

1. 基本命令
2. 文の区切りとブロック構造
3. 変数
4. 繰り返し
5. 条件分岐
6. 魔法 / 属性 / 合体魔法
7. ターゲット指定
8. プレイヤー / NPC 相互参照

- まずは、ステージ1〜4を成立させる**最小構文**を優先する
- 関数定義や高度な抽象化は、初期実装では無理に入れない
- パーサー実装しやすさよりも、**小学生にとって読みやすく直感的であること**を優先する
- ただし、ローカルヒントや AI ナビゲーターが説明しやすいよう、曖昧すぎる表現は避ける
