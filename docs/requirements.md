# Kotoba Quest 要件定義書

最終更新: 2026-04-13

---

## 1. プロジェクト概要

### 目的

子供たちの**発想力・創造性を育む**ことを主眼に置いた、プログラミング学習ゲーム。
既存サービスとの差別化は以下の通り。

| サービス | 特徴 | Kotoba Quest との差別化 |
|----------|------|------------------------|
| paiza | スキル判定・競技寄り | 正解を覚えるのではなく、自分で考えて試行錯誤する体験を重視 |
| Scratch | ビジュアルプログラミング寄り | テキストベースの論理的思考力を自然に習得させる |

### ターゲットユーザー

- **小中学生全体**
- 段階的な難易度設計により、プログラミング未経験者から中級者まで対応

## 3. ゲームシステム

### 3-1. ジャンル

ドラゴンクエスト的な**ターン制バトルRPG**。

### 3-2. バトル方式

**事前プログラミング型**

- バトル開始前にプレイヤーがコードを全部書く
- 書いたコードを「実行」するとバトルが始まり、結果が再生される
- バトル中にリアルタイムで操作することはない

### 3-3. プレイヤーリソース

| リソース | 説明 |
|----------|------|
| HP | ヒットポイント。0になるとゲームオーバー |
| MP | マジックポイント。魔法・特殊行動に消費する |

### 3-4. 難易度設計（魔法・MP）

#### 魔法属性

- 魔法属性は **火 / 水 / 雷 / 氷 / 風** の5属性とする
- プレイヤーが初期状態で使用できる属性は **火 / 水 / 雷** の3属性とする
- **氷 / 風** はステージクリア報酬として段階的に解放する

#### 合体魔法

- コード中で3属性以上の魔法を組み合わせた場合、その魔法を**合体魔法**として扱う
- 合体魔法は序盤から連発できないようにし、単属性魔法や2属性魔法も戦略として成立する難易度設計とする
- 序盤ステージでは、**3R後に合体魔法を撃てること**をバランスの基準とする
- 本仕様では、初期最大MPとラウンドごとのMP増加量により、**4R目開始時に初めて3属性合体魔法を使用可能**な設計を基準とする

#### MP仕様

- 初期ステージにおける戦闘開始時の最大MPは **50** とする
- 戦闘開始時の現在MPは、そのステージの初期最大MPと同値とする
- 各ラウンド開始時に、**最大MPを10増加**させる
- 各ラウンド開始時の現在MP回復量は、**その時点の最大MPの1/3** とする
- MP回復量の端数は**小数点以下切り捨て**とする
- 各ラウンド開始時は、**最大MPを10増加した後の値**を基準に現在MP回復量を計算する
- ステージをクリアしていくことで、戦闘開始時の初期最大MPを段階的に増加させる
- 戦闘開始時の初期最大MPは、最終的に **100** になる想定とする
- 例として、ステージ1では戦闘開始時の最大MPは **50**、2R目開始時は最大MP **60**・現在MP **20回復**、3R目開始時は最大MP **70**・現在MP **23回復** とする

#### 属性ごとの消費MP

- 1属性魔法の消費MPは **10** とする
- 2属性魔法の消費MPは **25** とする
- 3属性魔法の消費MPは **80** とする
- 4属性魔法の消費MPは **100** とする
- 5属性魔法の消費MPは **120** とする

#### 設計意図

- 序盤は単属性魔法と2属性魔法を中心に学習させる
- 3属性以上の合体魔法は強力な選択肢とするが、初期から最適解にならないようMP制約を設ける
- ステージ進行に応じて初期最大MPを増加させることで、プレイヤーに成長実感を与える
- 後半ステージでは、より複雑な属性組み合わせや高コスト魔法を扱えるようにする

### 3-5. ステージ構成と学習テーマ

- ステージ数は **6ステージ構成** を基本とする
- 学習順は、Pythonライクな実行モデルを意識し、**順次実行 -> 繰り返し -> 変数 -> 条件分岐 -> 総合課題 -> 高難易度総合課題** の流れを基本とする
- **氷属性** はステージ2のクリア報酬、**風属性** はステージ3のクリア報酬として解放する
- **全ステージをウェーブ制**とし、1ステージの中で段階的に敵や課題が変化する構成を基本とする
- 各ステージは、原則として **Wave1: 導入 / Wave2: 応用 / Wave3: まとめのボス** の3ウェーブ構成を基本とする

| ステージ | 学習テーマ | 主な学習内容 | 開始時に使用可能な属性 | クリア報酬 | 戦闘開始時最大MP | ウェーブ数 |
|----------|------------|--------------|------------------------|------------|------------------|------------|
| ステージ1 | 順次実行と単属性魔法の基本 | コードは上から順に実行されること、命令を書くと行動できること、単属性魔法で戦闘の基本を学ぶ | 火 / 水 / 雷 | 追加なし | 50 | 3 |
| ステージ2 | 繰り返しと行動の効率化 | 同じ行動を何度も書かずにまとめることで、コードを短くし、修正しやすくし、ミスを減らせることを学ぶ | 火 / 水 / 雷 | 氷 | 60 | 3 |
| ステージ3 | 変数・属性選択・協力コードの修正 | 値に名前を付けて扱う考え方を学び、氷属性を含めて状況に応じた魔法選択を試しながら、協力NPCのコードを読み解いて不具合を修正し、NPCの風魔法を活用して戦う | 火 / 水 / 雷 / 氷 | 風 | 70 | 3 |
| ステージ4 | 条件分岐と合体魔法の本格導入 | 敵の状態に応じて有効な属性を切り替えることで条件分岐を学び、道中でMPを溜めながら最終Waveで五属性合体魔法の爽快感を体験する | 火 / 水 / 雷 / 氷 / 風 | 追加なし | 80 | 3 |
| ステージ5 | 総合課題 | 新しい構文は増やさず、これまで学んだ順次実行、繰り返し、変数、条件分岐、属性選択、MP管理、合体魔法、NPC連携を自分で選んで組み合わせて攻略する | 火 / 水 / 雷 / 氷 / 風 | 追加なし | 90 | 3 |
| ステージ6 | ボスラッシュ総合振り返り | ステージ1〜5で登場したボスを5ウェーブ連続で相手にし、これまで学んだ要素を総復習する最終ステージとする | 火 / 水 / 雷 / 氷 / 風 | 追加なし | 100 | 5 |

- この配分により、ステージ1〜3では3属性合体魔法は戦闘中盤以降の選択肢となり、ステージ4以降では開幕から合体魔法を視野に入れた設計が可能になる

#### ウェーブ構成方針

- 各ステージは、**導入 -> 応用 -> まとめ** の流れで難易度が上がる3ウェーブ構成を基本とする
- ただし、**ステージ6のみ例外**として、ステージ1〜5の学習内容を総復習する**5ウェーブのボスラッシュ構成**とする
- Wave1では新しい概念を無理なく試せるようにし、Wave2ではその応用を求め、Wave3ではそのステージの学習内容をまとめて確認する
- 各ステージのWave3には、そのステージの学習テーマを象徴する**ボスまたはボス相当の強敵**を配置する
- ステージ6では、各Waveがそれまでのステージを代表するボス戦となり、段階的に既習要素を振り返る構成にする

| ステージ | Wave1 | Wave2 | Wave3 |
|----------|-------|-------|-------|
| ステージ1 | 弱い敵1体。攻撃のみを学ぶ導入戦 | 弱い敵2体。敵の攻撃と防御を見る | 少し強いボス1体。基本のまとめ |
| ステージ2 | 単純な敵1体。繰り返しを試し、コードが短くなる利点を体感する | 同じ対応が必要な敵2体。修正しやすさとミスの減少を学ぶ | 繰り返しを前提としたボス1体。長めの行動列を簡潔に書く価値を示し、クリア報酬として氷属性を解放 |
| ステージ3 | 氷属性を試せる敵1体。NPCとの連携導入を行い、NPCの風魔法の必要性を学ぶ | 属性選択とNPCコード修正を考える敵2体。NPCの風魔法を前提に攻略する | 変数・属性選択・NPC連携・NPCとの合体魔法を使わせるボス1体。クリア報酬として風属性を解放 |
| ステージ4 | 三属性の中から通る属性を見抜く敵1体 | 五属性の中から通る属性が異なる敵2体で条件分岐を使った消耗戦を行う | 2Wave目の敵4体と入力文字制限ボスを相手に耐えた後、五属性合体魔法で一掃する最終Wave |
| ステージ5 | 性質の違う敵2体を相手に優先順位を考える導入Wave | 状態変化と回復・強化を使う中ボス2体で総合判断を行う | 雑魚召喚や状態変化を使う総合ボス1体を、合体魔法の使いどころを見極めて攻略する |
| ステージ6 | 5ウェーブのボスラッシュ前半。ステージ1・2のボスで基礎を振り返る | ボスラッシュ中盤。ステージ3・4のボスで連携と条件分岐を振り返る | ボスラッシュ最終Wave。ステージ5の総合ボスで全体のまとめを行う |

#### ステージ1の敵構成

| ウェーブ | 敵構成 | 学ばせたいこと | 敵の想定行動 |
|----------|--------|----------------|--------------|
| Wave1 | 弱い敵1体 | コードを書くと攻撃できること、敵を倒す基本の流れ | 基本攻撃のみ |
| Wave2 | 弱い敵2体 | 複数の敵が登場すること、敵ごとに行動が異なること | 片方が攻撃、片方が防御 |
| Wave3 | 少し強いボス1体 | それまでに学んだ基本を使って少し長い戦闘を乗り切ること | 攻撃と防御を使い分ける |

- Wave1では、プレイヤーに最初の成功体験を与えるため、短いコードで確実に突破できる難易度にする
- Wave2では、敵も防御行動を取ることを見せ、プレイヤーに「相手の状態を見て行動を考える」導入を与える
- Wave3では、通常敵よりHPの高いボスを置き、ステージ1のまとめとして扱う
- ステージ1は、敵の行動説明そのものがチュートリアルになるよう、UI上で「攻撃」「防御」の意味を簡潔に表示できる設計が望ましい

#### ステージ2の敵構成

- ステージ2は、**同じ行動をまとめるメリットを明確に提示する繰り返し学習ステージ**とする
- プレイヤーには、単に`繰り返す`構文を教えるだけではなく、**コードが短くなる / 修正しやすくなる / 書き間違いを減らせる**という利点を体感させる
- 各Waveは、手書きで同じ命令を並べた場合の不便さが自然に見える敵構成にする
- ステージ2では、**長いコードほどダメージが通りにくい敵ギミック**を導入し、短く整理して書く必要性をゲーム上の挙動として提示する
- 文字数判定は、空白や改行の影響を受けにくいよう、**有効文字数ベース**で行うことを前提とする

| ウェーブ | 敵構成 | 学ばせたいこと | 敵の想定行動 |
|----------|--------|----------------|--------------|
| Wave1 | 弱い敵1体 | 同じ攻撃を何回も行う場面では、繰り返し構文を使うとコードを短くできること | 基本攻撃のみ |
| Wave2 | 同程度の敵2体 | 同じ処理をまとめて書くことで、回数や内容を後から修正しやすくなり、書き漏れや回数ミスを減らせること | 両方とも似た行動パターンを取り、長いコードへの耐性を持つ |
| Wave3 | HPが高めのボス1体 | 長い行動列をそのまま並べるより、繰り返し構文で整理した方が見通しが良く、強敵戦でも扱いやすいこと | 複数ターンにわたり一定の行動を続け、長いコードへの強い耐性を持つ |

- Wave1では、同じ攻撃を複数回行うだけで突破できる構成にし、`繰り返す(3)` のような最小構成で成功体験を与える
- Wave2では、似た対応を複数回求めることで、同じ命令を並べる書き方よりも、繰り返し構文の方が修正しやすく、さらに短いコードの方が有利になることを理解させる
- Wave3では、繰り返しを使わないとコード量が増えやすいボス戦にし、可読性と保守性の利点に加えて、短く整理したコードの方が高いダメージを出せることを定着させる
- ステージ2のUIやヒントでは、`同じ処理が続くならまとめて書ける`、`回数を変えるだけで調整できる` に加えて、`長いコードには耐性を持つ敵がいる` ことを短い文で明示することが望ましい

#### ステージ3の協力NPCとコード修正

- ステージ3では、プレイヤーと共に戦う**協力NPC**を登場させ、プレイヤー自身のコードだけでなく、他者のコードを読んで直す体験を学習テーマに含める
- プレイヤーのコードとNPCのコードは**別々のコード領域**として管理する
- プレイヤーコードとNPCコードは、お互いの値を**参照可能**とする
- 相互参照は、実装を単純化し学習難度を抑えるため、**読み取り専用**を基本とする
- 相手の値を参照する際は、ターン中の循環依存を避けるため、**ターン開始時点の値を参照する**前提とする
- ステージ3では、プレイヤーに最初から「バグがある」と直接通知するのではなく、**NPCの行動結果やセリフから違和感に気づかせる**演出を基本とする
- プレイヤーは、NPCのコードを確認し、意図と異なる箇所を修正することで戦闘を有利に進める
- ステージ3の不具合は、構文エラー中心ではなく、**動くが意図通りに機能しない論理的な不具合**を中心とする
- ステージ3では、**NPCのみが風属性魔法を使用可能**とし、プレイヤーはまだ風属性を直接使用できない
- ステージ3の敵には、**NPCの風魔法を使わないと倒しきれない特性**を持つものを登場させ、NPCとの連携が必須であることを明確にする
- ステージ3の最終ボスでは、プレイヤー単独ではなく、**NPCとの連携による合体魔法**を使うことで突破する構成とし、ステージ4で扱う合体魔法の学習へ接続する

| ウェーブ | 敵構成 | 学ばせたいこと | NPCコード上の不具合例 |
|----------|--------|----------------|------------------------|
| Wave1 | NPCの風魔法が有効な敵1体 | 協力NPCの存在を理解し、NPCの行動結果から違和感に気づくこと。プレイヤーだけでは突破できない場面があることを学ぶ | 変数の値が小さすぎて想定より弱い風魔法になる |
| Wave2 | 属性選択を考える敵2体 | プレイヤーとNPCが別コードで動きつつ、互いを参照して連携できること。NPCの風魔法を通すための準備を考えること | 参照する変数名や使用属性がずれていて、必要な風魔法が正しく発動しない |
| Wave3 | 連携を要求するボス1体 | NPCコードの複数箇所を見直し、変数・属性・連携の理解をまとめて使うこと。NPCとの合体魔法を体験し、次ステージへの導線にすること | 変数設定と属性選択の両方に軽い不具合があり、合体魔法の発動条件を満たせない |

- Wave1では、NPCが「風の魔法がうまく通らない…？」のように違和感を口にすることで、プレイヤーに不具合の存在を自然に気づかせる
- Wave2では、プレイヤーコードとNPCコードを見比べながら、どの値や属性を参照しているかを読む力を養い、NPCの風魔法を通すための連携を考えさせる
- Wave3では、NPCを正しく補助役として機能させ、NPCとの合体魔法を成立させないと攻略が難しい構成にし、コードを読む・直す・実行結果を確かめる流れを定着させる
- ステージ3のUIやヒントでは、`仲間のコードも読める`、`相手の値を見て連携できる`、`この敵には仲間の風魔法が必要` といった案内を出しつつ、`バグ` という言葉はNPCの気づきや会話を通じて段階的に理解させることが望ましい

#### ステージ4の条件分岐と合体魔法

- ステージ4では、**敵の状態によって有効な属性が切り替わる**ギミックを中心に据え、`もし` を使って状況に応じた属性選択を行うことを学ばせる
- プレイヤーには、毎回同じ魔法を使うのではなく、**敵状態を見て通る属性を選ぶ必要がある**ことを戦闘結果を通して理解させる
- ステージ4は3ウェーブ構成を維持しつつ、Wave1からWave2にかけて意図的に消耗感やストレスを高め、Wave3で**五属性合体魔法による一掃の爽快感**を強く出す構成とする
- 条件分岐の対象となる敵状態は、UI上で常に視認できる形で表示することを前提とする
- 敵状態と有効属性の対応は、プレイヤーが試行錯誤しつつ学べるようにしつつ、ヒントや演出で徐々に理解できる設計とする

| ウェーブ | 敵構成 | 学ばせたいこと | 敵ギミック |
|----------|--------|----------------|------------|
| Wave1 | 状態変化を行う敵1体 | 三属性のうち今通る属性を見抜き、条件分岐で使う属性を変える基本を学ぶ | 火 / 水 / 雷 のうちランダムで1属性だけが通り、選ばれた属性以外の2属性は吸収、氷 / 風は75%軽減される |
| Wave2 | 状態変化と防御行動を使う敵2体 | 2体の敵それぞれで通る属性が異なる状況を見ながら、どちらを優先するか、どの属性を使うかを条件分岐で判断することを学ぶ | 各敵が五属性のうちランダムで1属性だけを通し、残り属性を大きく軽減または無効化しつつ、防御や妨害も行う |
| Wave3 | Wave2の敵4体と文字数制限ボス1体が出現する最終Wave | 道中で学んだ条件分岐を使って耐えながらMPを溜め、入力文字数制限下でも五属性合体魔法で一気に決着をつけることを学ぶ | Wave2相当の敵4体が圧力をかける中、ボスが入力文字数を制限し、最終的に五属性合体魔法でまとめて殲滅できるタイミングが訪れる |

- Wave1では、火 / 水 / 雷 のうち通る属性がランダムで1つだけ選ばれる構成にし、`もし 敵が火状態なら 火属性を使う` のような最も基本的な条件分岐を自然に書かせる
- Wave1では、選ばれなかった残り2つの初期属性は吸収されて敵HPを回復し、氷 / 風は75%軽減されるため、正しい属性を選ばないと逆に不利になる設計とする
- Wave2では、2体の敵がそれぞれ別の有効属性を持つことで、単純なベタ書きでは対応しづらくし、条件分岐を使わないとジリジリ消耗する構成にする
- Wave2では、敵の防御行動や軽い妨害行動も加えることで、プレイヤーに「正しい属性を選んでも、撃つタイミングや対象を考えないと苦しい」というストレスを与える
- Wave3では、Wave2相当の敵を4体まで増やして圧力を高めつつ、ボスが入力文字数制限をかけることでコードの自由度も奪い、最後に**五属性合体魔法でまとめて吹き飛ばせる**ようにして、そこまでの我慢が報われる爽快感を演出する
- Wave3の入力文字数制限は、Stage2で学んだ「短く書く」意識を再び要求するギミックとして機能させる
- ステージ4の最終Waveは、道中のストレスを解放する演出が重要であり、五属性合体魔法の発動時には敵を一掃する視覚効果や演出を強めに入れることが望ましい
- ステージ4のUIやヒントでは、`今の状態で通る属性`、`状態が変わるタイミング`、`五属性合体魔法の発動条件が近いこと` などを段階的に示し、理不尽ではなく「苦しいが耐えれば気持ちよく勝てる」体験にすることが望ましい

#### ステージ5の総合課題

- ステージ5は、**新しい構文を教える場ではなく、これまで学んだ要素を自分で選んで組み合わせる総合課題**とする
- ステージ5では、`繰り返し / 変数 / 条件分岐 / 属性選択 / MP管理 / 合体魔法 / NPC連携` を総合的に使わせる
- プレイヤーに単一の正解だけを押し付けるのではなく、**複数の攻略方針が成立する設計**を目指す
- ただし、何を見て判断すべきかは UI やヒントで伝わるようにし、理不尽ではなく「自分で解けた」と感じられる難易度にする
- ステージ5は Stage6 よりもヒントを多めに残し、**考えれば解ける総合問題**として位置づける

| ウェーブ | 敵構成 | 学ばせたいこと | 敵ギミック |
|----------|--------|----------------|------------|
| Wave1 | 性質の違う敵2体 | どちらを優先して倒すべきか、どの属性を使うべきかを判断すること | 片方は属性相性が重要で、もう片方は防御や妨害を行う |
| Wave2 | 中ボス2体 | 状態変化・回復・強化を見ながら、ターゲット優先度、条件分岐、MP管理をまとめて考えること | 1体は状態変化で弱点が変わり、もう1体は回復や強化を行う |
| Wave3 | 総合ボス1体 | これまで学んだ要素をすべて使い、合体魔法を撃つべきタイミングまで含めて判断すること | 雑魚召喚、状態変化、防御、強攻撃準備などを使い、合体魔法のタイミングを見極める必要がある |

- Wave1では、敵の役割が明確に異なる構成にし、プレイヤーに「まず何を処理すべきか」を自分で考えさせる
- Wave2では、敵の状態変化と回復・強化を同時に扱わせることで、単純なベタ書きではなく、条件分岐と優先順位の判断が必要になる構成にする
- Wave3では、雑魚召喚や強攻撃準備など複数の要素を重ねつつ、合体魔法は「撃てる時に撃つ」のではなく、**撃つべきタイミングを見極めて使う**ことを学ばせる
- ステージ5では、プレイヤー主体で押し切る戦い方、NPCと連携して安定を取る戦い方など、**複数の攻略方針が成立する**ことが望ましい
- ステージ5のUIやヒントでは、`どの敵が危険か`、`どの敵が回復や強化をしているか`、`今が合体魔法の好機か` といった判断材料を見える形で提示することが望ましい
- ステージ5のクリア体験は、Stage6 のボスラッシュ総復習に入る前の「ここまで学んだことを使って勝てた」という達成感を与えるものにする

#### ステージ6のボスラッシュ総合振り返り

- ステージ6は、**ステージ1〜5で登場した各ボスを順番に振り返る5ウェーブのボスラッシュ**とする
- ステージ6では、新しい文法や新システムは導入せず、**これまで学んだことを総合的に使えるかを確認する最終復習ステージ**とする
- 各Waveでは、それまでのステージの学習テーマを象徴するボスを再登場させ、単なる再戦ではなく、最終ステージにふさわしい緊張感を持たせるために**強化版または再構成版**として登場させる
- ステージ6全体を通して、プレイヤーに「ここまでの学びを全部使って勝ち抜く」感覚を与えることを重視する

| ウェーブ | 振り返るステージ | 敵構成 | 学ばせたいこと |
|----------|------------------|--------|----------------|
| Wave1 | ステージ1 | ステージ1のボスの強化版1体 | 攻撃・防御の基本と、戦闘の流れを最初に思い出させる |
| Wave2 | ステージ2 | ステージ2のボスの強化版1体 | 繰り返し構文による短く効率的なコード設計を再確認する |
| Wave3 | ステージ3 | ステージ3のボスの強化版1体 | NPC連携、コード修正、協力による攻略を振り返る |
| Wave4 | ステージ4 | ステージ4のボスの強化版1体 | 条件分岐、属性切り替え、合体魔法の使いどころを再確認する |
| Wave5 | ステージ5 | ステージ5の総合ボスの強化版1体 | ここまでのすべての要素を使った最終確認として攻略する |

- Wave1では、最初のステージを象徴する攻撃・防御の基本を短く思い出させ、ボスラッシュの導入とする
- Wave2では、長いコードに対する不利や、短く整理したコードの強さを再認識させることで、ステージ2の学びを復習させる
- Wave3では、NPCとの連携が成立しないと突破しづらい構成にし、仲間のコードを見る・直す・連携する流れを再確認させる
- Wave4では、敵状態に応じた属性切り替えや、合体魔法を撃つタイミングの判断を再び求めることで、ステージ4の応用内容を振り返らせる
- Wave5では、ステージ5の総合ボスをより強化した形で登場させ、優先順位、条件分岐、MP管理、NPC連携、合体魔法まで含めた総仕上げを行う
- ステージ6の演出は、`懐かしいが強くなって戻ってきたボスたち` という印象を与え、プレイヤーが成長を実感できる構成にすることが望ましい
- ステージ6のクリア体験は、「難しかったが、ここまで覚えたことを全部使って勝ち切れた」という最終的な達成感を与えるものにする

## 6. 技術要件

### 6-1. 採用技術

| カテゴリ | 技術 | 用途 / 備考 |
|----------|------|-------------|
| アプリ基盤 | Next.js App Router | 画面構成、ルーティング、サーバー / クライアントの責務分離を行う |
| 言語 | TypeScript | フロントエンド、ゲームロジック、API 層を統一して実装する |
| ホスティング | Vercel | Next.js との親和性が高く、無料枠での運用やプレビュー環境の利用を想定する |
| バトル描画 | PixiJS v8 | バトル演出、敵状態変化、合体魔法、ボスラッシュなどの視覚表現を実装する |
| コードエディタ | CodeMirror 6 | 独自言語エディタ、ハイライト、行番号、補助UIの土台として利用する |
| 状態管理 | Zustand | React UI とバトル状態、実行状態、ステージ進行状態の接続に利用する |
| ゲームロジック | TypeScript モジュール | パーサー出力の実行、敵行動、ダメージ計算、Wave進行などを React から分離して実装する |
| 言語パーサー | Ohm.js | 独自日本語言語の構文解析、AST 生成、実行前バリデーションを行う |
| API 層 | Next.js Route Handlers | AI パートナー連携、将来のセーブAPI、設定保存APIの受け口として利用する |
| AI連携 | Claude API（Anthropic） | パートナーキャラクターの応答生成、コードへの助言、文脈に応じたヒント提示に利用する |
| ストリーミング | Server-Sent Events（SSE） | AI パートナーの応答を段階的に表示する場合の伝送方式として利用する |
| 単体 / 結合テスト | Vitest | パーサー、AST、ゲームロジック、計算処理、APIユーティリティの検証に利用する |
| E2Eテスト | Playwright | ステージ進行、コード入力、戦闘結果、AI UI を含むユーザー導線の検証に利用する |
| APIモック | MSW | 開発時およびテスト時に AI API や保存 API をモックする |

> **注意:** 現プロトタイプは Vite + Peggy 構成だが、本実装は Next.js / Ohm.js 前提で進める。

### 6-2. 実装方針

- Next.js App Router を正式なアプリ基盤とし、画面UIと API を同一リポジトリで管理する
- バトルシーンは PixiJS を用いたクライアント側描画とし、React は周辺UIや画面状態の制御を担当する
- 独自言語エディタは CodeMirror 6 を利用し、**実際の正誤判定や実行処理の正本は Ohm.js パーサー**とする
- パーサー、スペル実行、敵AI、ダメージ計算、Wave進行などのゲームロジックは、React 依存を持たない TypeScript モジュールとして実装する
- Zustand は UI 状態とゲームエンジン状態の橋渡しに利用し、描画ロジックや計算ロジックそのものはストアに持ち込みすぎない構成を基本とする
- AI パートナー連携はクライアントから直接呼ばず、Next.js Route Handlers 経由で Claude API を呼び出す
- AI 応答を段階表示したい場合は SSE によりストリーミングし、子どもにとって待ち時間が長く感じにくい体験を目指す
- プレイヤーコードと NPC コードのような複数コード領域は、エディタUI上では分離して見せつつ、内部的には共通のパーサー / 実行基盤で扱う

### 6-3. テスト方針

- パーサーの文法、AST、独自言語の実行結果、ダメージ計算、属性相性、MP計算は Vitest により単体テストを行う
- ルートハンドラや AI 連携アダプタなどのサーバー側処理も Vitest の対象とする
- ステージ開始からクリアまでの導線、エディタ入力、戦闘演出との接続、AI UI を含むユーザー導線は Playwright で E2E テストを行う
- 外部API依存を避けるため、開発時やテスト時の API 応答は MSW によりモックできる構成を前提とする
